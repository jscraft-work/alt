# 02. Domain Model

## 1. 문서 목적과 범위

이 문서는 [spec2.md](/Users/jongsoobae/workspace/alt-java/spec2.md)와 [01-system-architecture.md](/Users/jongsoobae/workspace/alt-java/docs/01-system-architecture.md)를 구현 관점으로 연결하는 도메인 모델 기준선이다. 목적은 "어떤 테이블을 만들지"보다 먼저 "시스템이 어떤 개념을 어떤 경계로 다뤄야 하는지"를 고정하는 데 있다.

포함 범위:

- 전략 템플릿과 전략 인스턴스의 개념 경계
- 트레이딩 사이클, 판단, 주문, 포트폴리오의 핵심 모델
- 프롬프트/모델/파라미터의 상속과 판단 시점 설정값 규칙
- 수집 데이터의 도메인 분리와 소비 방식
- audit/ops 개념과 운영 보조 상태
- 구현 단계에서 반드시 지켜야 할 불변조건과 비즈니스 규칙

비범위:

- 물리 테이블, 컬럼 타입, 인덱스, FK 이름
- JPA 매핑 방식
- 조회 최적화를 위한 read model 상세 구조
- 외부 API 요청/응답 DTO 상세

## 2. 모델링 원칙

- 도메인 모델은 구현 규칙과 상태 전이를 표현한다.
- DB 스키마는 이 모델을 저장하기 위한 수단이며, 동일 개념이 여러 테이블로 분해될 수 있다.
- 전략 인스턴스는 운영 단위이고, 템플릿은 정의 단위다.
- 트레이딩 사이클은 설정을 소비하는 실행 단위이며, 제품 DB에는 사이클 실행 로그와 판단 로그를 함께 남긴다.
- 글로벌 수집 데이터와 인스턴스 종속 데이터는 강하게 분리한다.
- `auto_paused`는 lifecycle의 주상태가 아니라 운영 보조 상태다.
- 설정 저장과 실행 반영은 분리한다. 저장은 즉시, 실행 반영은 다음 사이클부터다.

## 3. 도메인 범위와 하위 서브도메인

핵심 서브도메인은 다음과 같이 본다.

- `strategy`
  - 전략 템플릿, 전략 인스턴스, 감시 종목 구성, 설정 상속 규칙
- `trading`
  - 사이클 실행, 판단 생성, 주문 의사결정, reconcile, auto-paused 판정
- `portfolio`
  - 인스턴스 단위 자산, 현금, 보유 종목, 손익 계산
- `marketdata`
  - 시장 스냅샷, 1분봉, 장중 입력용 시세 참조
- `news`
  - 뉴스 원문 메타, 요약, 유용성 판정
- `disclosure`
  - DART 공시와 종목/DART 기업코드 연결
- `macro`
  - 일일 매크로 스냅샷
- `ops`
  - audit log, 운영 이벤트, 사이클 실행 로그, 상태 계산 입력

## 4. Aggregate 경계

### 4.1 StrategyTemplate

전략 템플릿은 새 인스턴스를 만드는 기준 정의다.

책임:

- 전략명, 설명, 기본 실행 의도 정의
- 템플릿 기본 파라미터와 기본 모델 설정 제공
- 신규 인스턴스 생성 시 초기값의 출처 역할 수행

템플릿은 운영 기록을 직접 가지지 않는다. 판단/주문/자산은 항상 인스턴스에 귀속된다.

### 4.2 StrategyInstance

전략 인스턴스는 실제 운영되는 핵심 기준 객체다.

책임:

- lifecycle 상태 보유
- 실행 모드(`paper/live`), 전략 예산, 연결 계좌, 인스턴스별 override 보유
- 인스턴스 종속 프롬프트/모델/파라미터/감시 종목의 현재 구성 참조
- auto-paused 보조 상태와 해제 조건 관리
- 사이클 실행 로그, 판단, 주문, 포트폴리오의 상위 소유자 역할

다른 대부분의 운영 기록은 `instance_id`를 기준으로 귀속된다.

### 4.3 PromptSet

프롬프트는 인스턴스 종속 설정이며 버전 이력을 가진다.

책임:

- 트레이딩 판단 프롬프트의 현재 활성 버전 관리
- 새 버전 생성과 이전 버전 재로드 지원
- 판단 시작 시점 설정값의 원천 제공

프롬프트는 단일 트레이딩 판단 프롬프트로 운영한다. 필요하면 본문 안에서 보유 종목 검토와 신규 후보 검토를 함께 구조화할 수 있지만, 판단 결과는 항상 단일 응답과 `orders[]`로 생성된다.

템플릿 프롬프트는 버전 이력을 두지 않는 "복제용 현재 기본값"이고, 인스턴스 생성 이후에는 인스턴스 프롬프트가 독립적으로 버전 이력을 가진다.

### 4.4 ModelProfile

모델 설정은 용도별 모델 선택의 현재 상태를 표현한다.

예시 용도:

- 트레이딩 판단 모델
- 일일 리포트 모델
- 뉴스 요약/판정 모델

모델 설정은 인스턴스별 또는 템플릿 기본값으로 존재할 수 있으나, 트레이딩 사이클이 소비하는 것은 항상 인스턴스 해석 결과다.

### 4.5 ParameterPolicy

시스템 파라미터는 전역 기본값을 관리하는 정책 객체다.

책임:

- 글로벌 파라미터 유지
- 실행 시 공통으로 참조할 전역 기준 제공

템플릿과 인스턴스의 개별 실행 파라미터는 각각 `default_execution_config`, `execution_config_override` 안에서 관리한다.

### 4.6 Watchlist

감시 종목은 인스턴스별 운영 입력 집합이다.

책임:

- 글로벌 종목 마스터를 참조해 인스턴스별 종목 집합 유지
- 매핑 단위 추가/소프트 삭제
- 사이클 입력 대상 종목 확정

종목 자체는 글로벌 자산 식별자이고, 감시 여부는 인스턴스 정책이다.

### 4.7 TradeCycleLog

사이클 실행 로그는 실제로 시작된 한 번의 트레이딩 사이클을 기록하는 핵심 기록 객체다.

책임:

- 사이클 시작/종료 시각 보유
- 현재 또는 마지막 단계 상태 보유
- 실패 지점과 실패 사유 보유
- 판단 로그 생성 여부 추적
- 같은 사이클의 주문/포트폴리오 갱신 흐름 추적의 기준점 제공

상태값:

- `READY`
- `RECONCILE_PENDING`
- `INPUT_LOADING`
- `LLM_CALLING`
- `DECISION_VALIDATING`
- `ORDER_PLANNING`
- `ORDER_EXECUTING`
- `PORTFOLIO_UPDATING`
- `COMPLETED`
- `FAILED`

이 로그는 실제로 시작된 사이클만 남긴다. 락 획득 실패, 시간대 밖, 이미 실행 중이라 시작되지 못한 스킵은 운영 이벤트로만 남긴다.

### 4.8 TradeDecisionLog

판단 로그는 한 번의 LLM 판단 결과를 기록하는 핵심 기록 객체다.

책임:

- 사이클 실행 로그와의 연결 보유
- 원시 응답과 파싱 결과 연결
- 판단 상태(`EXECUTE` / `HOLD` / `FAILED`) 보유
- 사이클 수준 `summary`, `rationale`, `confidence` 보유
- 주문 제안 목록 `orders[]` 보유
- 판단 시점 설정값과 비용/토큰 정보 보유
- 생성된 주문 의도 및 실제 주문과의 연결 보유
- 판단 시점의 프롬프트, 모델, 전략 입력 스펙, 인스턴스 실행 설정, 감시 종목 목록을 함께 저장

판단 로그는 주문이 없더라도 성립할 수 있다. 예를 들어 `HOLD`가 그렇다. 반대로 구조적 또는 의미적 검증에 실패한 경우에는 판단 전체가 `FAILED`로 기록된다.

사이클 실행 로그와 판단 로그의 관계는 `1:0..1`이다. reconcile 실패처럼 판단 단계에 진입하기 전에 끝난 사이클은 판단 로그 없이 종료될 수 있다.

### 4.9 TradeOrderIntent / TradeOrder

구현에서는 주문 제안, 내부 실행 계획, 실제 브로커 주문 기록을 분리하는 편이 안전하다.

- `TradeOrderIntent`
  - LLM이 제안한 주문안을 시스템이 검증한 내부 실행 계획
  - `symbol`, `side`, `quantity`, `order_type`, `price`를 포함한 실행 단위
  - 주문별 `rationale`과 `evidence[]`를 함께 보유
- `TradeOrder`
  - 외부 브로커 또는 paper 체결 엔진에 제출된 주문 기록
  - 상태머신과 체결 누적 수량을 가짐

`paper`에서는 일반적으로 LLM 주문안 1건이 실제 주문 1건으로 바로 이어진다. `live`에서는 주문 거부, 부분체결, 재주문 분기 가능성이 있으므로 `TradeOrderIntent -> N TradeOrder` 구조를 기본으로 본다.

### 4.10 Portfolio

포트폴리오는 인스턴스 단위 자산 상태 aggregate다.

구성:

- 현금 잔고
- 총 자산가치
- 오늘 실현손익
- 보유 종목 목록

보유 종목은 별도 entity 집합으로 관리하되, 일관성 기준은 인스턴스 포트폴리오에 둔다.
현재 포트폴리오는 별도 현재 상태 테이블로 관리한다. 주문 직후 포트폴리오 상태는 각 `trade_order`에 함께 저장하고, 모든 주문/체결 이력을 매번 재생해 현재 상태를 계산하지는 않는다.

### 4.11 BrokerAccount

연결 계좌는 live 인스턴스가 참조하는 외부 계좌 식별자다.

책임:

- 브로커 계좌 식별 정보 관리
- 활성 live 인스턴스와 시점상 1:1 제약 유지
- 마스킹/표시용 메타 정보 제공

연결 계좌 변경 이력은 별도 entity로 두지 않고 audit log로만 남긴다.

### 4.12 AssetMaster

글로벌 종목 마스터는 종목 식별과 외부 코드 매핑의 기준이다.

책임:

- 종목코드, 종목명, 시장 구분
- DART 기업코드 매핑
- 비활성/숨김 상태 관리

이 객체는 트레이딩 판단의 직접 소유자가 아니라, 여러 도메인이 참조하는 기준 데이터다.

## 7. 핵심 관계

핵심 관계는 아래처럼 해석한다.

- `StrategyTemplate 1 -> N StrategyInstance`
- `StrategyInstance 1 -> N WatchlistRelation`
- `AssetMaster 1 -> N WatchlistRelation`
- `StrategyInstance 1 -> N PromptVersion`
- `StrategyInstance 1 -> N TradeDecisionLog`
- `TradeDecisionLog 1 -> N TradeOrderIntent`
- `TradeOrderIntent 1 -> N TradeOrder`
- `StrategyInstance 1 -> 1 current Portfolio`
- `StrategyInstance 1 -> N OpsEvent`
- `BrokerAccount 1 <-> 1 active live StrategyInstance (temporal)`

추가 해석 규칙:

- 판단 로그 1건에는 주문 제안이 0건 이상 들어갈 수 있다.
- 하나의 주문 의도는 정확히 하나의 판단에 속한다.
- 하나의 실제 주문은 정확히 하나의 주문 의도에서 파생된다.
- 보유 종목은 인스턴스 간 공유되지 않는다.
- 뉴스/공시/시장/매크로 데이터는 글로벌 데이터이며 인스턴스 FK를 기본 전제로 하지 않는다.

## 8. 템플릿, 인스턴스, 설정 상속

### 8.1 개념 분리

- 템플릿은 신규 인스턴스 생성의 기본 출처다.
- 인스턴스는 실제 실행 설정의 소유자다.
- 일부 설정은 "복제"되고, 일부는 "상속 해석"된다.

### 8.2 복제되는 것

인스턴스 생성 시 다음 항목은 템플릿 또는 기존 인스턴스에서 복제된다.

- 프롬프트 초기 버전
- 감시 종목 초기 집합
- 모델 설정 초기값
- 파라미터 기본값 또는 override
- 실행 모드, 예산, 연결 계좌 같은 운영 기본값

복제 이후에는 원본과 독립적으로 변경된다. 특히 프롬프트는 자동 전파되지 않는다.

### 8.3 상속 해석되는 것

시스템 파라미터는 다음 우선순위로 해석된다.

1. 글로벌 기본값
2. 템플릿 기본값
3. 인스턴스 override

이 규칙은 "현재 유효 파라미터"를 계산하기 위한 도메인 서비스 또는 정책 객체가 책임져야 한다.

### 8.4 인스턴스 유효 구성

실행 시점에서 인스턴스가 실제로 사용하는 구성은 다음 합성 결과다.

- 인스턴스가 가진 현재 활성 프롬프트 버전
- 인스턴스 모델 설정 또는 템플릿 기본 모델
- 파라미터 상속 해석 결과
- 활성 감시 종목 집합

이 합성 결과가 판단 로그에 함께 저장될 판단 시점 설정값의 원천이 된다.

## 9. 판단 시점 설정값

### 9.1 왜 필요한가

설정은 운영자가 언제든 바꿀 수 있지만, 사이클은 시작 시점 기준으로 일관되게 실행되어야 한다. 따라서 판단과 주문은 "당시 어떤 설정과 어떤 입력 데이터를 사용했는지"를 추적할 수 있어야 한다.

### 9.2 판단 로그에 함께 저장할 설정값

판단 로그에는 당시 다음 설정을 함께 저장한다.

- 프롬프트 버전
- 전략 입력 스펙
- 모델 선택
- 해석 완료된 시스템 파라미터
- 감시 종목 집합
- 실행 모드

이 설정값은 판단 로그 내부의 불변 기록이다.

### 9.3 기록 목적

판단 상세에서 중요한 것은 별도 스냅샷 테이블의 존재가 아니라 다음 성질이다.

- 판단 시점에 사용된 정확한 프롬프트 원문을 재현할 수 있어야 한다.
- 모델명과 모델 버전 또는 별칭 해석 결과를 재현할 수 있어야 한다.
- 주기, 비용 한도, 수수료 등 파라미터의 유효값을 재현할 수 있어야 한다.

### 9.4 입력 데이터 기준값

트레이딩 사이클은 설정만이 아니라 입력 데이터 기준시각도 고정해야 한다.

- 시장 스냅샷 기준시각
- 분봉 조회 범위
- 뉴스/공시 조회 컷오프
- 매크로 스냅샷 일자
- 장중 일시적 호가 스냅샷 참조

호가는 영속 aggregate가 아니라 일시 입력이므로, 원본은 2일만 보존한다. 장기 보존은 판단 로그 원문과 메타 정보로 대신하며, 최소한 입력 시각과 출처는 남겨야 한다.

## 10. 인스턴스 lifecycle과 auto_paused 보조 상태

### 10.1 주상태

인스턴스 lifecycle 주상태는 다음 세 가지다.

- `draft`
- `active`
- `inactive`

`draft`, `active`, `inactive` 사이 전이는 모두 허용한다.

### 10.2 상태 의미

- `draft`
  - 생성 직후 상태
  - 검토와 편집이 가능한 준비 상태
- `active`
  - 스케줄러가 실행 대상으로 취급하는 운영 상태
- `inactive`
  - 실행은 멈추지만 기록과 포트폴리오는 보존되는 상태

### 10.3 auto_paused 보조 상태

`auto_paused`는 별도 lifecycle 상태가 아니라 이유와 해제조건을 갖는 운영 보조 상태로 둔다.

예시 사유:

- `reconcile_failed`

도메인 의미:

- lifecycle이 `active`여도 `auto_paused`이면 신규 사이클 실행 또는 신규 주문 제출이 제한된다.
- `auto_paused`는 운영 알림과 시스템 상태 카드의 입력이다.
- `auto_paused` 해제는 자동 또는 수동 규칙을 가진다.

### 10.4 auto_paused와 lifecycle의 조합 규칙

- `draft`와 `inactive`는 `auto_paused` 대상이 아니다.
- `active + auto_paused`는 "운영 대상으로 등록돼 있으나 실행 제한 중"을 의미한다.
## 11. 판단, 주문, 포트폴리오 관계

### 11.1 TradeDecisionLog

판단 로그는 최소 다음 정보를 가진다.

- 판단 상태 `EXECUTE` / `HOLD` / `FAILED`
- 사이클 수준 요약 또는 근거
- confidence
- 에러 여부
- 원시 요청/응답 참조
- 토큰 사용량과 추정 비용
- 주문 제안 목록
- 판단 시점 설정값

`HOLD`는 빈 주문 배열만으로 암묵 처리하지 않는다. 사이클 수준 상태로 명시해야 한다.

### 11.2 TradeDecisionLog와 TradeOrder

판단은 실제 주문으로 변환될 수 있고, 변환되지 않을 수도 있다.

주문 미발생 예:

- `HOLD`
- 파싱 실패
- 정책 검증 실패

주문 발생 시 규칙:

- 한 판단은 여러 주문 의도를 포함할 수 있다.
- 각 주문 의도는 실제 주문 여러 건으로 분할 가능하다.
- 각 주문 의도는 하나의 판단에만 속한다.
- `client_order_id`는 주문마다 새로 생성

### 11.3 TradeOrder 상태머신

주문 상태는 다음 집합을 가진다.

- `requested`
- `accepted`
- `partial`
- `filled`
- `canceled`
- `rejected`
- `failed`

구현은 허용 전이를 명시적으로 제한해야 한다. 예를 들어 `filled`에서 다시 `accepted`로 돌아갈 수 없다.

### 11.4 Portfolio와 Position

포트폴리오는 인스턴스 단위로 독립 유지된다.

- 동일 종목을 여러 인스턴스가 각자 보유 가능
- 평단은 이동가중평균
- 평가손익/실현손익은 spec 계산 규칙을 따름

`Position`은 포트폴리오 내부 entity로 보는 것이 자연스럽다. 외부에서 직접 수정하는 대신 주문 체결 또는 reconcile 결과를 통해 갱신돼야 한다.

## 12. Collector 관점 데이터 도메인

collector는 도메인이 아니라 데이터 공급 파이프라인이다. 소비자는 collector 구현이 아니라 아래 데이터 도메인에 의존해야 한다.

### 12.1 marketdata

포함 대상:

- 현재가 또는 시장 스냅샷
- 1분봉
- 장중 시세 기준시각

도메인 규칙:

- 글로벌 데이터다.
- 차트, 대시보드, 트레이딩 사이클이 함께 소비한다.
- 호가는 장기 영속 모델이 아니라 단기 캐시 입력이다.

### 12.2 news

포함 대상:

- 뉴스 메타데이터
- 기사 링크
- 요약
- 유용성 상태 `useful / not_useful / unclassified`에 해당하는 도메인 값

뉴스는 글로벌 데이터지만, 조회 시 인스턴스 감시 종목으로 빠른 필터링할 수 있다.
뉴스 유용성 판정에 사용하는 모델은 글로벌 설정으로 관리한다.

### 12.3 disclosure

포함 대상:

- 공시 제목
- 발행 시각
- 본문 미리보기
- 외부 원문 링크
- DART 기업코드 연결

공시는 뉴스와 별도 도메인으로 유지한다. 수집 주기, 원천, 매핑 방식이 다르기 때문이다.

### 12.4 macro

포함 대상:

- 환율
- 해외 지수
- 금리
- 그 외 일일 보조 거시 지표

매크로는 트레이딩 입력의 보조 자료이며, 종목 단위 데이터가 아니라 일자 기준 글로벌 스냅샷이다.

## 13. Audit와 Ops 개념

### 13.1 AuditLog

audit log는 변경 이력의 사실 기록이다.

필수 성질:

- append-only
- actor 포함
- 대상 entity 종류와 식별자 포함
- 변경 전/후 요약 포함
- 영구 보존

대상 사건 예:

- 인스턴스 생성, 복제, 상태 전환
- 프롬프트 버전 생성/롤백
- 감시 종목 변경
- 모델 설정 변경
- 시스템 파라미터 변경
- 인증 사건

### 13.2 OpsEvent

운영 이벤트는 시스템 보호와 장애 감지를 위한 이벤트다.

예시:

- rate limit 초과
- 외부 API 장애
- 웹소켓 지연/끊김
- 로그인 차단

audit와 차이:

- audit는 "누가 무엇을 바꿨는가"
- ops event는 "운영 중 무엇이 발생했는가"

## 14. 불변조건과 비즈니스 규칙

구현 시 반드시 강제해야 할 핵심 규칙은 다음과 같다.

### 14.1 인스턴스와 템플릿

- 모든 인스턴스는 정확히 하나의 템플릿을 가진다.
- 템플릿 수정은 기존 인스턴스에 자동 전파되지 않는다.
- 인스턴스 복제는 설정만 복제하며 자산/이력은 복제하지 않는다.
- 복제된 인스턴스의 초기 lifecycle은 항상 `draft`다.

### 14.2 lifecycle

- 하드 삭제는 제공하지 않는다.
- `draft`와 `inactive`에서만 인스턴스명, 실행 모드, 전략 예산, 연결 계좌를 변경할 수 있다.
- `active` 상태 편집은 다음 사이클부터만 반영된다.

### 14.3 live/paper

- `live` 모드는 연결 계좌 없이는 활성화할 수 없다.
- 시점상 하나의 활성 live 인스턴스만 하나의 연결 계좌를 사용할 수 있다.
- `paper`와 `live`는 같은 `TradeOrder` 상태머신을 공유한다.
- `paper`는 부분체결을 시뮬레이션하지 않으며, 일반적인 정상 흐름은 `requested -> accepted -> filled`다.
- `paper`에서는 일반적으로 `TradeOrderIntent`와 `TradeOrder`가 1:1에 가깝고, `live`에서는 1:N이 될 수 있다.

### 14.4 사이클 실행

- 한 인스턴스의 사이클은 직렬 실행돼야 한다.
- 이전 사이클이 끝나지 않았으면 다음 주기는 스킵된다.
- 사이클은 시작 시점에 확정된 판단 시점 설정값을 사용해야 한다.
- 사이클 도중 저장된 설정 변경은 현재 사이클에 영향을 주지 않는다.

### 14.5 판단과 주문

- 판단 JSON은 사이클 수준 상태와 주문 제안 목록 계약을 만족해야 한다.
- `HOLD`는 명시적 사이클 상태여야 하며, `orders[]`가 비어 있더라도 정상 판단으로 기록될 수 있다.
- 주문은 판단 없이 존재할 수 없다.
- 주문 상태 전이는 정의된 상태머신을 벗어날 수 없다.
- reconcile 실패 시 인스턴스는 `auto_paused(reason=reconcile_failed)`로 전환된다.
- 판단 출력이 구조적 또는 의미적으로 깨지면 일부 주문만 살리지 않고 판단 전체를 `FAILED` 처리한다.

### 14.6 비용/상한

- 활성 인스턴스 수는 시스템 하드 상한 10을 넘을 수 없다.
- 인스턴스별 일일 LLM 비용 상한은 추정 비용 합산 기준 운영 알림 용도로 사용한다.

### 14.7 데이터 도메인

- 뉴스/공시/시장/매크로 데이터는 글로벌 데이터다.
- 감시 종목은 글로벌 종목 마스터 참조 + 인스턴스 매핑으로 표현한다.
- 글로벌 종목 마스터는 하드 삭제하지 않는다.
- 호가는 영속 도메인 entity가 아니다.

## 16. 03 DB 스키마와 04 API 명세에 주는 직접적 기준

이 문서는 후속 문서에 다음 기준을 제공한다.

- DB 스키마는 `StrategyInstance`, `TradeDecisionLog`, `TradeOrder`, `AuditLog`, `OpsEvent`를 중심으로 정규화해야 한다.
- API 명세는 인스턴스 현재 구성과 판단 로그에 저장된 판단 시점 설정값을 구분해서 노출해야 한다.
- 설정 API는 "현재 저장값"과 "다음 사이클 반영값"의 개념 차이를 드러내야 한다.
- 매매이력 API는 판단 상세에 판단 시점 설정값, 주문 연결, 토큰/비용 메타를 포함해야 한다.
- 뉴스/공시/차트 API는 글로벌 데이터와 인스턴스 빠른 필터를 분리해야 한다.

## 17. 문서 가정

- 구현 스택은 Java 21 + Spring Boot 3.x, PostgreSQL, Redis, React + TypeScript + Vite로 고정한다.
- 시각 저장과 업무 날짜 기준은 KST로 고정하며, 날짜 계산에는 `business_date`를 사용한다.
- 트레이딩 판단은 LLM 기반이지만, 도메인 모델은 특정 LLM 벤더나 subprocess 구현에 종속되지 않도록 유지한다.

# ALT-Web 기획서

## 1. 문서 목적

이 문서는 ALT-Web의 제품 범위와 화면 동작을 기획서 수준으로 정리한 문서다. 개발자, 기획자, 디자이너가 동일한 기준으로 기능 범위와 운영 규칙을 공유할 수 있도록 모순·모호·누락을 줄이는 것을 목표로 한다.
대상은 기획자, 디자이너, 프론트엔드/백엔드 개발자이며, 구현 세부와 세금/수수료 정산 공식은 범위에서 제외한다.

## 2. 제품 정의

ALT-Web은 한국 주식 주기형 트레이딩 전략의 운영 콘솔이다.
사용자는 전략 인스턴스별 자산 상태, 매매 판단 로그, 뉴스/공시, 분봉 차트를 한곳에서 확인하고, 운영자는 전략 템플릿과 실행 인스턴스를 관리해 다음 트레이딩 사이클에 반영할 수 있다.

본 문서에서 전략은 크게 두 층위로 다룬다.

- 전략 템플릿: 프롬프트, 모델, 파라미터, 기본 감시 종목 구성을 담는 설계 단위
- 전략 인스턴스: 실제로 스케줄에 따라 실행되는 운영 단위

## 3. 목표와 비목표

### 3.1 목표

- 전략별 트레이딩 현황을 빠르게 모니터링할 수 있어야 한다.
- 새로운 전략을 운영자가 쉽게 추가할 수 있어야 한다.
- 새로운 전략은 모두 동일한 주기형 실행 구조 위에서 손쉽게 복제·조정할 수 있어야 한다.
- 같은 전략 템플릿을 `paper`/`live` 등 서로 다른 모드의 인스턴스로 손쉽게 운영할 수 있어야 한다.
- LLM의 판단 결과와 실제 주문 결과를 나란히 추적할 수 있어야 한다.
- 뉴스, 공시, 시세 데이터를 같은 운영 화면 안에서 탐색할 수 있어야 한다.
- 운영자가 웹에서 전략, 감시 종목, 프롬프트, 모델, 시스템 파라미터를 관리할 수 있어야 한다.

### 3.2 비목표

- 브로커 HTS/MTS를 대체하는 범용 수동 주문 UI 제공
- 다중 운영자/권한 그룹 관리
- 회원가입, 비밀번호 재설정, 리프레시 토큰
- 세무 신고, 회계 장부, 정산 공식 설명
- **이벤트 트리거형 즉시 매매 엔진 운영** (이벤트 드리븐 아키텍처 자체를 배제하지는 않으며, 이벤트 정보는 §8.2의 주기형 판단 입력값으로 흡수한다)
- **투자 자문/추천 제공** (모든 표시 데이터는 운영 모니터링 정보일 뿐, 투자 권유로 해석되지 않는다)

## 4. 사용자와 권한

### 4.1 사용자 유형

| 유형 | 설명 |
|------|------|
| 비로그인 사용자 | 대시보드, 매매이력, 뉴스·공시, 차트를 조회할 수 있다. |
| 운영자 | 로그인 후 설정 변경과 운영용 API 사용이 가능하다. |

본 서비스는 외부에 일부 공개되는 모니터링 페이지로 운영한다. 따라서 운영자 콘솔이지만 비로그인 조회는 의도적으로 허용한다.

### 4.2 인증 정책

- 인증 방식은 단일 운영자 ID/PW + JWT다.
- refresh token은 사용하지 않는다.
- JWT는 `HttpOnly + Secure + SameSite` 쿠키로 전달한다.
- 세션은 sliding 방식으로 연장한다. 기본 idle timeout은 8시간, absolute max session은 7일로 둔다.
- 인증된 요청 시 남은 세션 시간이 짧으면 새 JWT를 재발급해 쿠키를 갱신한다(세부 임계값은 하위 설계에서 정의).
- 운영자 계정은 DB의 `app_user`로 관리한다. 초기 계정 생성 방식은 하위 설계에서 정의한다.
- 로그인 실패가 같은 IP에서 5회 누적되면 5분 동안 차단한다. 차단/카운트는 신뢰 가능한 reverse proxy 체인 기반의 원본 client IP를 기준으로 한다(체인 파싱 규칙은 하위 설계 문서에서 정의).
- 차단 카운트는 최근 실패 시도 윈도(5분)를 기준으로 하며, 성공 로그인 시 해당 IP의 실패 카운트는 초기화된다.
- 운영자가 IP 차단을 즉시 해제할 수 있는 수단을 제공한다(구체 수단은 하위 설계).
- 상태 변경 요청에는 CSRF 방어를 적용한다(세부 방식은 하위 설계 문서에서 정의).
- 인증이 필요한 요청이 만료되면 로그인 페이지로 이동시키되, 대시보드 30초 자동 갱신 중 401은 예외로 처리한다.
  - 대시보드 30초 자동 갱신 중 401: 자동 갱신만 중지하고 인라인 알림으로 표시한다(전체 화면 redirect 회피).
- 자동 트레이딩 사이클과 내부 수집 프로세스의 인증/권한은 운영자 웹 세션과 분리한다.

## 5. 핵심 사용 시나리오

### 5.1 일상 모니터링

1. 사용자는 상단 전략 선택기에서 전체 또는 특정 전략 인스턴스를 선택한다.
2. 특정 전략 인스턴스 선택 시 대시보드에서 자산, 보유종목, 시스템 상태, 최근 활동을 본다.
3. 전체 선택 시 전략 인스턴스별 요약 카드 목록을 보고 관심 인스턴스로 진입한다.

### 5.2 판단 검증

1. 사용자는 매매이력 화면에서 주문 이력 또는 판단 로그를 선택한다.
2. 기간, 결과, 종목 기준으로 필터링한다.
3. 판단 상세에서 LLM 요청 프롬프트, 응답 원문, 파싱 결과, 연결 주문, 사용 모델 및 토큰/비용을 확인한다.

### 5.3 데이터 탐색

1. 사용자는 뉴스·공시 화면에서 종목과 기간을 기준으로 검색한다.
2. 뉴스는 유용/비유용/미판단 상태로 좁혀 볼 수 있다.
3. 차트 화면에서는 선택한 날짜의 종목별 1분봉과 주문 시점을 함께 확인한다.

### 5.4 운영 설정

1. 운영자는 설정 화면에서 전략 템플릿을 만들고, 이를 기준으로 실행 인스턴스를 생성하거나 활성/비활성 상태를 바꾼다.
2. 새 인스턴스는 기존 템플릿 또는 기존 인스턴스를 기준으로 쉽게 복제할 수 있어야 한다(설정만 복제, 자산/이력은 복제하지 않는다).
3. 인스턴스별 감시 종목을 추가/삭제한다.
4. 트레이딩 판단 프롬프트를 수정하고 이전 버전을 다시 불러올 수 있다.
5. 모델 설정, 시스템 파라미터, 실행 모드를 조정하고 다음 사이클부터 반영한다.

## 6. 화면 구조

### 6.1 공통 구조

- 상단 네비게이션: `대시보드`, `매매이력`, `뉴스·공시`, `차트`, `설정`
- 좌측 상단: 로고와 전략 인스턴스 선택기
- 우측 상단: 테마 토글, 로그인/로그아웃
- 모바일: 햄버거 메뉴 제공
- 정의되지 않은 라우트 접근 시 404 화면 + "대시보드로 돌아가기" 링크를 제공한다.

### 6.2 라우트

| 경로 | 화면 | 비로그인 접근 | 비고 |
|------|------|----------------|------|
| `/` | 대시보드 또는 전략 오버뷰 | 가능 | |
| `/trades` | 매매이력 | 가능 | |
| `/news` | 뉴스·공시 | 가능 | |

## 7. 화면별 요구사항

### 7.0 화면별 인스턴스 의존성 요약

| 화면 | 전략 인스턴스 의존 | 전체(글로벌) 동작 |
|------|--------------------|--------------------|
| 대시보드 | 인스턴스 의존 | 전체 선택 시 전략 오버뷰로 fallback |
| 매매이력 | 인스턴스 의존(허용) | 전체 선택 시 `인스턴스명` 컬럼 함께 표시 |
| 뉴스·공시 | 글로벌 | 특정 인스턴스 선택 시 감시종목 빠른 필터 제공 |
| 차트 | 글로벌 | 특정 인스턴스 선택 시 감시종목 기본 선택, 전체 시 글로벌 마스터에서 수동 선택 |

### 7.1 대시보드

목적: 현재 전략 인스턴스의 운영 상태를 한 화면에서 파악한다.

주요 동작:

- 30초 주기로 자동 갱신한다(장 시간과 무관하게 동작).
- 수동 새로고침 버튼과 마지막 갱신 시각을 KST 기준으로 제공한다.
- 전략 인스턴스가 선택되어 있지 않으면 전략 오버뷰로 대체한다.
- 30초 자동 갱신 중 인증 401 응답이 오면 자동 갱신을 중지하고 인라인 알림으로 안내한다(전체 화면 redirect 회피).
- 외부 데이터 소스(source) 장애로 갱신에 실패하면 마지막 성공값을 유지한 채 "마지막 갱신 N분 전"과 "신뢰도 낮음" 배지를 함께 표시한다.

구성:

- 헤더 정보
  - 템플릿명
  - 인스턴스명
  - 실행 모드 `paper/live`
  - 연결 계좌 식별 정보가 있으면 함께 표시(live 계좌번호는 마스킹)
- 요약 카드
  - 총 자산가치
  - 현금 잔고
  - 오늘 실현손익
  - 오늘 거래 횟수
- 보유종목 테이블
  - 종목명, 종목코드, 수량, 매수단가(이동가중평균), 현재가, 평가손익, 수익률
- 시스템 상태 카드
  - 트레이더, 시장 데이터, 뉴스 수집, DART 공시, 웹소켓의 상태 표시
  - 상태는 정상/지연/중단 3단계로 구분(임계값은 §7.1.1)
  - 마지막 동작 시각을 툴팁으로 KST로 제공
  - 사이클 직렬 실행 정책상 이전 사이클이 미종료되어 다음 주기를 스킵한 경우 "사이클 지연" 표시
  - 당일 트레이딩 판단 요약 제공
- 최근 활동
  - 최근 주문 목록
  - 최근 에러가 있으면 별도 노출

#### 7.1.1 시스템 상태 임계값

| 서비스 | 정상 | 지연 | 중단 | 비고 |
|--------|------|------|------|------|
| 웹소켓 | ≤ 30초 | ≤ 5분 | > 5분 | 장중에만 적용. 장외에는 "휴장/장외" 표시 |
| 시장 데이터 | ≤ 30초 | ≤ 5분 | > 5분 | 장중에만 적용. 장외에는 "휴장/장외" 표시 |
| 트레이더 사이클 | ≤ 실행주기 + 90초 | ≤ 2 × (실행주기 + 90초) | > 2 × (실행주기 + 90초) | |
| 뉴스 수집 | ≤ 수집주기 + 5분 | ≤ 3 × (수집주기 + 5분) | > 3 × (수집주기 + 5분) | |
| DART 공시 | ≤ 수집주기 + 5분 | ≤ 3 × (수집주기 + 5분) | > 3 × (수집주기 + 5분) | |

### 7.2 전략 오버뷰

목적: 여러 전략 인스턴스를 한 번에 비교하고 진입한다.

주요 동작:

- 전략 선택기가 `전체`일 때 노출한다.
- 기본적으로 활성 전략 인스턴스를 카드로 보여준다.
- `active`이면서 `auto_paused_reason != null`인 인스턴스도 카드에 포함하며, "AUTO-PAUSED" 배지와 사유를 함께 표시한다.
- 단, 비활성 인스턴스라도 미청산 보유가 남아 있으면 예외적으로 카드에 포함한다.
- 카드 클릭 시 해당 전략 인스턴스 대시보드로 이동한다.
- 비활성 인스턴스 카드에는 "미청산 보유 있음" 경고를 표시한다.

카드 정보:

- 템플릿명, 인스턴스명
- 실행 모드 `paper/live`
- 총 자산
- 오늘 손익
- 현재 보유 종목
- 오늘 BUY/SELL/HOLD 요약

### 7.3 매매이력

목적: 실제 주문 결과와 LLM 판단 기록을 조회한다.

탭 구성:

- 주문 이력
- 판단 로그

공통 규칙:

- 기본 조회 기간은 최근 7일이다.
- 페이지네이션은 20건 단위다.
- 전략 인스턴스, 기간, 종목, 결과 조건으로 필터링한다.
- 자동 갱신은 적용하지 않으며 수동 새로고침으로 대신한다.
- 인스턴스 전체 선택 시 결과에는 `인스턴스명` 컬럼이 함께 노출된다.

주문 이력에서 보여줄 정보:

- 주문 시각 (KST)
- 종목명/종목코드
- BUY/SELL
- 주문가, 수량, 금액
- 손익, 수익률
- 주문 상태(`requested / accepted / partial / filled / canceled / rejected / failed`)
- 누적 체결 수량 (부분체결 케이스)

판단 로그에서 보여줄 정보:

- 판단 시각 (KST)
- 판단 상태(`EXECUTE` / `HOLD` / `FAILED`)
- 평가 대상 범위 요약 (`held_only` / `full_watchlist`)
- 대표 종목/주문 요약
- 에러 여부
- 판단 근거를 함께 표시한다. `FAILED`에는 실패 사유를 함께 표시한다.
- 사용 모델, 입력 토큰, 출력 토큰, 추정 비용

판단 상세:

- LLM 요청 프롬프트 원문
- LLM 응답 원문
- 파싱된 판단 JSON (사이클 단위 상태 + `orders[]` + `evidence[]`)
- 연결된 주문 정보 (1:N 분할 허용)
- 판단 시점 설정값
  - 프롬프트 버전 식별자
  - 모델 식별자와 호출 파라미터
  - 전략 입력 스펙 확정본
  - 인스턴스 실행 설정 확정본
  - 감시 종목 확정 목록
- 구조적 또는 의미적 검증 실패 시 부분 수용하지 않고 해당 사이클 판단 전체를 실패로 기록

### 7.4 뉴스·공시

목적: 트레이딩 판단에 사용되는 외부 정보 데이터를 탐색한다.

탭 구성:

- 뉴스
- 공시

공통 규칙:

- 기본 조회 기간은 최근 7일이다.
- 페이지네이션은 20건 단위다.
- 종목과 기간 필터를 제공한다.
- 자동 갱신은 적용하지 않는다(수동 새로고침).
- 인스턴스 선택 시 해당 인스턴스의 감시종목 빠른 필터를 함께 제공한다.

뉴스 탭:

- 제목, 관련 종목명, 발행 시각, 요약, 유용성 상태 표시
- 뉴스 1건이 여러 종목과 연결될 수 있으며, 화면에서는 하나의 row 안에 관련 종목명을 함께 표시한다.
- 기사 원문은 요약/유용성 판단 입력으로 수집할 수 있으나, 화면에는 본문 미리보기 대신 요약 중심으로 표시한다.
- 유용성 필터: 전체 / 유용 / 비유용 / 미판단
- 외부 기사 링크로 이동 가능

공시 탭:

- 제목, 종목명, 발행 시각, 본문 미리보기 표시
- 외부 공시 링크로 이동 가능

### 7.5 차트

목적: 종목별 장중 흐름과 주문 타이밍을 함께 본다.

주요 동작:

- 기본 날짜는 오늘(KST)이다.
- 선택한 날짜의 1분봉을 종목별 카드로 보여준다.
- 종목 체크박스로 표시 대상을 다중 선택할 수 있다.
- 동시 표시 상한은 8개이며, 초과 선택은 차단하고 안내한다. 한 카드 안에 여러 종목을 series로 오버레이하는 방식은 본 spec 범위 외다.
- 데스크톱 2~3열 / 모바일 1열 그리드로 정렬한다. 모바일에서는 차트 본문 가독성을 우선한다.
- 인스턴스 선택 시 해당 인스턴스의 감시종목을 기본 선택으로 제공하고, 전체 선택 시 글로벌 종목 마스터에서 수동으로 선택한다.
- 오늘 날짜이면서 장중이면 1분 주기 자동 갱신 상태를 표시한다(장외에는 자동 갱신을 적용하지 않는다).
- 주문 이력이 있으면 매수/매도 시점을 차트 위에 표시한다.

### 7.6 설정

목적: 운영자가 시스템 입력값을 웹에서 관리한다.

공통 정책:

- `/settings` 자체가 로그인 필요 화면이다(§4.2, §6.2).
- 저장하지 않은 변경사항이 있으면 탭 이동/새로고침 전에 경고한다.
- 저장 후 변경사항은 다음 사이클부터 반영된다(현재 진행 중인 사이클은 시작 시점 설정값을 그대로 사용).
- 동일 항목을 여러 운영 세션이 동시에 수정한 경우 낙관적 잠금(optimistic lock, `updated_at` / `version` 기반)을 적용해 충돌 시 최신 상태로 다시 불러와 재시도하게 한다.
- 같은 전략 템플릿으로 여러 인스턴스를 만들 수 있어야 한다.

탭 구성:

- 전략 템플릿 관리
  - 템플릿 생성
  - 기존 템플릿 복제 또는 기본형 템플릿 생성
  - 설명 입력
  - 기본 실행 주기 설정 (1분 ~ 30분, 권장 5분)
- 전략 인스턴스 관리
  - 템플릿 기준 인스턴스 생성 (생성 시 항상 `draft` 상태로 시작)
  - 기존 인스턴스 복제 (설정만 복제, 자산/보유/판단/주문 이력은 복제하지 않음)
  - 인스턴스명 입력 (`draft / inactive`에서만 변경 가능)
  - 실행 모드 선택 `paper / live` (`draft / inactive`에서만 변경 가능)
  - 전략 예산 입력 (`draft`에서만 변경 가능)
  - 연결 계좌 선택 (`draft / inactive`에서만 변경 가능; live 모드는 연결 계좌 필수)
  - 활성/비활성 전환 — 상태머신은 §10.2 참조
- 종목 마스터 관리
  - 종목코드, 종목명, DART 기업코드 관리
  - 종목코드 입력 시 DART 기업코드 자동 lookup을 기본으로 하고, lookup 실패 또는 모호한 결과에서는 운영자 수동 입력/검색 선택을 허용한다.
  - 글로벌 마스터의 하드 삭제는 제공하지 않으며 비활성/숨김과 이력 보존만 허용한다.
- 감시 종목 설정
  - 인스턴스별 감시 종목 추가/삭제
  - 감시 종목 수 자체에는 spec 차원의 상한을 두지 않는다.
  - 글로벌 종목 마스터를 참조해 인스턴스별 감시 종목 목록을 관리한다.
  - 인스턴스 매핑 삭제는 소프트 삭제로 처리한다.
- 프롬프트 설정
  - 템플릿 기본 프롬프트 관리
  - 인스턴스 트레이딩 판단 프롬프트 편집 (인스턴스 생성 시 템플릿에서 복사)
  - 필수 변수 목록 제공
  - 클릭 삽입 지원
  - 이전 버전 로드 지원 (인스턴스별 독립 버전 이력)
  - 저장 전 영향 경고 표시
  - 템플릿 프롬프트 수정은 신규 인스턴스에만 적용되며 기존 인스턴스에는 자동 전파되지 않는다.
  - 선택적으로 종목별 인사이트/know-how 메모를 프롬프트에 주입할 수 있어야 하며, 기본값은 비활성이다.
- 모델 설정
  - 트레이딩 모델: 템플릿 기본 모델 선택 + 인스턴스별 개별 설정
  - 리포트 모델: 글로벌 운영 설정에서 선택
  - 뉴스 모델: 글로벌 운영 설정에서 선택
- 전략 입력 스펙
  - 템플릿 기본 입력 스펙 관리
  - 인스턴스별 입력 스펙 개별 설정 관리
  - 평가 대상 범위 선택 `held_only / full_watchlist`
  - 글로벌 데이터 출처(`source`)별 포함 여부 선택
  - 데이터 출처별 과거 조회 범위(`lookback`) / 최대 항목 수 설정
  - 데이터 소스별 설정 대상 예: 뉴스, 공시, 시장 스냅샷, 분봉, 매크로, 장중 호가 스냅샷
- 인스턴스 실행 설정
  - 실행 주기 (인스턴스 단위, 1 ~ 30분, 권장 5분)
  - 인스턴스별 일일 LLM 비용 상한
  - 실행 모드와 운영 예산의 현재값 확인
- 글로벌 운영 설정
  - 시장 스냅샷 수집 주기
  - 뉴스 수집 주기
  - DART 수집 주기
  - 사이클 실행 시작/종료 시각 (KST)
    - 전략 인스턴스의 판단 사이클을 장중 어느 시간대까지 실행할지 정한다.
    - 예: 장 마감 직전 노이즈를 피하기 위해 종료 시각을 15:20 등으로 조정할 수 있다.
  - 뉴스 수집 건수
    - 1회 수집 실행당 최대 수집 건수
  - 활성 인스턴스 상한 (권장 3 / 하드 10)
  - 기본값 초기화
    - 글로벌 운영 설정 탭의 기본값 묶음만 초기화한다.

## 8. 운영 흐름

### 8.1 자동 수집 및 판단

시스템은 별도 프로세스로 활성 전략 인스턴스를 기준으로 다음 작업을 반복 수행한다.

- 트레이딩 사이클 실행
- 시장 스냅샷 수집
- 뉴스 수집 및 요약
- DART 공시 수집
- 웹소켓 기반 분봉 적재

### 8.2 주기형 전략 운영 원칙

전략 템플릿은 모두 동일한 주기형 트레이딩 구조를 따른다.

- 장중 일정 주기마다 반복 판단한다.
- 판단 주기는 인스턴스별로 1분 ~ 30분 사이에서 설정하며 권장 기본값은 5분이다.
- 정보 반영 최대 지연은 `설정 주기 + 평균 사이클 처리시간`이다.
- 한 인스턴스의 사이클은 직렬로 실행한다. 이전 사이클이 종료되지 않은 시점에 다음 주기가 도래하면 해당 주기는 스킵하고 시스템 상태에 "사이클 지연"으로 표시한다.
- 전략은 "프롬프트 + 입력 스펙"으로 정의한다. 어떤 데이터를 가져올지는 프롬프트 문구가 아니라 입력 스펙 설정으로 결정한다.
- 입력 스펙은 글로벌 데이터 출처(`source`)별 포함 여부와 과거 조회 범위(`lookback`) / 최대 항목 수를 지정하고, 평가 대상 범위를 `held_only` 또는 `full_watchlist`로 고정한다.
- 뉴스, 공시, 시세, 분봉, 장중 호가 스냅샷, 매크로 등은 입력 스펙에 따라 같은 사이클 안에서 함께 평가한다.
- 이벤트성 정보(뉴스/공시 등)는 즉시 매매로 이어지지 않고 주기형 판단의 입력값으로 흡수된다. 이는 의도된 선택이며, 즉시성보다 일관성을 우선한 결과다.
- 사이클 시작 시점의 프롬프트, 모델, 전략 입력 스펙, 인스턴스 실행 설정, 감시 종목 목록을 고정해 사용하며, 사이클 도중 변경된 설정은 다음 사이클부터 반영한다.
- LLM은 목표 비중만 제안하는 것이 아니라, 각 사이클에서 실행 가능한 주문 제안(`orders[]`)까지 직접 산출한다.
- 시스템은 LLM이 제안한 주문안을 그대로 신뢰하되, 주문 제출 직전 최소 안전검증만 수행한다.
  - 예: 양수 수량 여부, `order_type`/`price` 조합 일관성, 매수 가능 현금 범위, 매도 가능 보유 수량 범위
  - 위 검증에 실패하면 해당 사이클의 주문 제출은 수행하지 않고 검증 실패로 기록한다.
- 실제로 시작된 사이클은 인스턴스별 `사이클 실행 로그` 엔티티로 저장한다. 상태값은 `READY / RECONCILE_PENDING / INPUT_LOADING / LLM_CALLING / DECISION_VALIDATING / ORDER_PLANNING / ORDER_EXECUTING / PORTFOLIO_UPDATING / COMPLETED / FAILED`를 사용한다.
- 판단 로그는 인스턴스별 단일 엔티티로 저장하며 상태는 `EXECUTE / HOLD / FAILED`를 사용한다.
- `사이클 실행 로그 ↔ 판단 로그`는 `1:0..1` 관계다. reconcile 실패처럼 판단 단계에 진입하지 못한 사이클은 판단 로그 없이 끝날 수 있다.
- 각 판단 로그에는 재현 가능한 추적을 위해 같은 시점의 설정값을 함께 저장한다.
- 판단 로그 안의 설정값에는 프롬프트 버전 식별자, 모델 식별자와 호출 파라미터, 전략 입력 스펙 확정본, 인스턴스 실행 설정 확정본, 감시 종목 확정 목록을 포함한다.
- `FAILED`는 LLM 호출 또는 판단 결과 처리까지 진입했지만 구조적/의미적 검증, 입력 구성, 내부 처리에 실패한 경우를 뜻한다.
- 락 획득 실패, 실행 시간대 밖, 이미 진행 중인 사이클 때문에 시작 자체가 되지 않은 스킵은 `사이클 실행 로그`를 만들지 않고 애플리케이션 로그와 운영 모니터링 이벤트로만 남긴다.

#### 8.2.1 외부 API 장애 처리

- 공통(시장 데이터 장애):
  - 시세/분봉/장중 호가 등 판단 입력에 필요한 시장 데이터 조회가 실패하면 해당 턴의 사이클 실행 로그와 판단 로그를 `FAILED`로 기록하고 신규 주문을 발생시키지 않는다.
  - 입력 데이터 검증 또는 내부 처리에 실패한 경우도 사이클 실행 로그와 판단 로그를 `FAILED`로 기록한다.
  - 외부 API 장애나 내부 처리 오류가 발생하면 운영 알림을 즉시 발생시킨다.
  - 같은 인스턴스에서 같은 필수 시장 데이터 소스 실패가 반복되면 시스템 상태 카드에는 최근 장애 이력을 누적해 표시한다.
  - 시장 데이터 장애만으로 인스턴스를 자동 일시중지(`auto_paused`)로 전환하지는 않는다.
- live 인스턴스:
  - 계좌 조회 API 실패 시 자산 정보를 오래된 값(stale)으로 표시하고 신규 주문을 차단한다.
  - 주문 API 실패 시 해당 판단을 "주문 실패"로 기록하고 같은 사이클 안에서 즉시 재시도하지 않는다. 미확정 주문은 다음 사이클 진입 전 주문 상태 동기화(reconcile) 절차에서 처리한다.
- paper 인스턴스:
  - 브로커 계좌/주문 API는 사용하지 않는다.
  - 다만 시장 데이터 장애는 paper 판단 입력과 평가가 계산에 직접 영향을 주므로 공통 정책을 동일하게 적용한다.

기획 원칙:

- 새 전략 추가 시 새로운 엔진을 만드는 대신 기존 주기형 전략 템플릿의 설정, 프롬프트, 종목 구성을 복제해서 시작할 수 있어야 한다.
- 같은 템플릿으로 여러 인스턴스를 만들 수 있어야 하며 인스턴스 간 차이는 실행 모드, 연결 계좌, 예산, 실행 주기 등 운영 설정으로 표현할 수 있어야 한다.
- 전략 추가 작업은 운영자가 UI에서 완료할 수 있어야 하며 코드 수정이 기본 경로가 되어서는 안 된다.
- 전략 성과는 동일한 주기형 구조 안에서 템플릿별, 인스턴스별로 비교 가능해야 한다.

### 8.3 웹에서 다루는 운영 개념

- 전략 템플릿은 판단 로직과 기본 설정을 담는 정의 단위다.
- 전략 템플릿의 핵심 구성은 트레이딩 판단 프롬프트와 전략 입력 스펙이다.
- 전략 인스턴스는 독립된 자본, 상태, 실행 모드를 가진 운영 단위다.
- 같은 템플릿에서 `paper` 인스턴스와 `live` 인스턴스를 각각 운영할 수 있어야 한다.
- 인스턴스별 감시 종목을 관리할 수 있다.
- 사이클 실행 로그는 실제 사이클 진행 단계와 실패 지점을 추적 가능해야 한다.
- 판단 로그는 판단 상태, 판단 근거, 주문 제안을 함께 추적 가능해야 한다.
- 주문 이력은 판단이 제안한 `orders[]`와 실제 실행 결과를 인스턴스 기준으로 연결 추적 가능해야 한다.
- 데이터 수집기가 지연되거나 중단되면 대시보드에 드러나야 한다.

## 9. 데이터 범위

### 9.1 핵심 데이터 항목

문서 관점에서 관리하는 핵심 데이터는 다음과 같다.

- 전략 템플릿
- 전략 인스턴스
- 전략 입력 스펙 (템플릿 기본값 + 인스턴스별 개별 설정)
- 종목 마스터 (글로벌)
- DART 기업코드 참조 (글로벌)
- 인스턴스별 감시 종목 매핑
- 종목별 인사이트/know-how 메모 (글로벌 종목 마스터 종속)
- 자산 및 보유 종목 (인스턴스 단위)
- 사이클 실행 로그 (인스턴스 단위)
- 판단 로그 (인스턴스 단위)
- 주문 이력 (인스턴스 단위)
- 시장 스냅샷 (글로벌)
- 1분봉 데이터 (글로벌)
- 뉴스 (글로벌)
- 공시 (글로벌)
- 매크로 스냅샷 (글로벌)
- 연결 계좌

추가로 운영용 데이터로 다음을 관리한다.

- 운영 이벤트 로그
- audit log (영구 보존)
- 일일 리포트용 집계 데이터

### 9.2 entity 관계

| 관계 | 카디널리티 | 비고 |
|------|------------|------|
| 전략 템플릿 ↔ 전략 인스턴스 | 1:N | 템플릿 수정은 신규 인스턴스에만 영향 |
| 전략 템플릿 ↔ 전략 입력 스펙(기본값) | 1:1 | 템플릿 생성 시 기본 입력 스펙을 가진다 |
| 전략 인스턴스 ↔ 전략 입력 스펙 개별 설정 | 1:0..1 | 없으면 템플릿 기본값을 그대로 상속 |
| 전략 인스턴스 ↔ 자산 집계 | 1:1 | 인스턴스 단위 분리 추적 |
| 전략 인스턴스 ↔ 보유 종목 | 1:N | 동일 종목을 여러 인스턴스가 독립 보유 가능 |
| 전략 인스턴스 ↔ 사이클 실행 로그 | 1:N | 실제로 시작된 사이클만 기록 |
| 사이클 실행 로그 ↔ 판단 로그 | 1:0..1 | 판단 단계 진입 전 실패할 수 있음 |
| 전략 인스턴스 ↔ 판단 로그 | 1:N | `EXECUTE / HOLD / FAILED` 단위 기록 |
| 판단 로그 ↔ 주문 이력 | 1:N | 분할 주문 허용 |
| 종목 마스터 ↔ 인스턴스별 감시 종목 매핑 | 1:N | 글로벌 마스터 + 인스턴스 매핑 분리 |
| 전략 인스턴스 ↔ 감시 종목 매핑 | 1:N | |
| 종목 마스터 ↔ 종목별 인사이트/know-how 메모 | 1:0..1 | 메모는 글로벌 종목 마스터에 종속되며 여러 인스턴스에서 공유 가능 |
| 뉴스 ↔ 종목 마스터 | N:M | 뉴스 1건이 여러 종목과 연결될 수 있으며 종목도 여러 뉴스와 연결될 수 있다 |
| 연결 계좌 ↔ 활성 live 인스턴스 | 시점상 1:1 | 시간축상 한 계좌가 여러 인스턴스와 매핑된 이력은 가능 |
| DART 기업코드 참조 ↔ 종목 마스터 | 1:N | 각 종목은 하나의 DART 기업코드에 연결되며 하나의 기업코드는 여러 종목이 공유할 수 있다 |

### 9.3 entity 정책

- 종목 마스터는 글로벌 단위로 관리한다. 하드 삭제는 제공하지 않으며 비활성/숨김 처리 + 이력 보존만 허용한다.
- 종목 마스터는 종목코드, 종목명과 함께 하나의 DART 기업코드 참조를 가진다. 동일 기업의 보통주/우선주 등 여러 종목이 같은 DART 기업코드를 공유할 수 있다.
- 인스턴스별 감시 종목은 매핑 단위로 추가/소프트 삭제한다.
- 종목별 인사이트/know-how 메모는 글로벌 종목 마스터 종속 데이터로 관리하며, 인스턴스별 복제본을 두지 않는다. 프롬프트 주입 여부만 인스턴스 설정으로 제어한다.
- 판단 로그에는 판단 시점 설정값을 함께 저장하며 이후 수정하지 않는다.
- 판단 시점 설정값은 최소 다음을 포함한다: 프롬프트 버전 식별자, 모델 식별자와 호출 파라미터, 전략 입력 스펙 확정본, 인스턴스 실행 설정 확정본, 감시 종목 확정 목록.
- 과거 판단 재현은 판단 로그에 저장된 설정값과 LLM 요청·응답 원문을 기준으로 수행한다.
- 사이클 실행 로그는 실제로 시작된 사이클만 저장한다. 락 획득 실패, 시간대 밖, 이미 실행 중인 인스턴스의 스킵은 운영 이벤트로만 남긴다.
- 뉴스는 본문 entity와 종목 매핑 entity를 분리해 관리하며, 뉴스 1건은 여러 종목과 연결될 수 있다.
- 호가는 장기 영속 entity로 관리하지 않는다. 사이클 입력용 단기 캐시로만 다루며 §8.2의 "장중 호가 스냅샷"은 사이클 내 일시적 입력값을 의미한다.
- 호가 원본은 단기 보존만 하며, 장기 추적은 판단 로그에 남는 프롬프트/응답 원문을 기준으로 한다.
- 매크로 스냅샷은 환율/해외지수/금리 등을 일일 1회 수집한 글로벌 entity로, 1년 보존하며 사이클 LLM 입력의 보조 자료로 사용한다.
- audit log는 변경 이벤트를 영구 보존한다. audit log를 위한 별도 화면은 본 spec 범위 외이며 API만 제공한다.
  - 대상: 인스턴스 생성/복제/상태 전환/설정 수정, 프롬프트 버전, 글로벌 종목 마스터 변경, 감시 종목 매핑 변경, 시스템 파라미터 변경, 모델 설정 변경, 인증 사건.
  - 필드: 발생 시각(KST), actor(운영자 ID 또는 시스템), 대상 entity 종류와 식별자, 변경 전/후 요약.
- 사이클 시작/종료/실패 단계 자체는 `사이클 실행 로그`에 남기고, 스킵/수집 지연/자동 일시중지(`auto_paused`) 전환 같은 운영 이벤트는 애플리케이션 로그와 모니터링 이벤트로 함께 기록한다.
- 운영 이벤트 로그는 외부 API 장애, LLM 호출 실패, 수집 지연, 자동 일시중지(`auto_paused`) 전환 등을 기록하며 §10.9 알림의 입력으로 사용한다.

### 9.4 데이터 보존 정책

본 보존 정책은 spec에 고정한다(시스템 파라미터로 조정하지 않으며 변경은 spec 개정으로 처리한다).

| 데이터 | 즉시 조회 보존(hot) | 보관 저장(cold/archive) | 비고 |
|--------|----------|--------------|------|
| 1분봉 | 90일 | — | |
| 시장 스냅샷 | 30일 | — | |
| 장중 호가 스냅샷 원본 | 2일 | — | 사이클 입력용 단기 보존만 허용 |
| 뉴스 / 공시 | 1년 | — | |
| 사이클 실행 로그 | 90일 | 이후 보관 저장(cold archive)으로 옮겨 영구 보존 | |
| 판단 로그 / 주문 이력 / 현재 자산 상태 | 영구 | — | |
| 감시 종목 매핑 이력 / 인스턴스 프롬프트 버전 이력 | 90일 | 이후 보관 저장(cold archive)으로 옮겨 영구 보존 | |
| LLM 요청·응답 원문 | 90일 | 이후 보관 저장(cold archive) 또는 압축 보존 | 요약만 남기는 식의 손실 보존은 금지 |
| 매크로 스냅샷 | 1년 | — | |
| audit log | — | 영구 | |
| 운영 이벤트 로그 | — | 영구 | |

- 판단 로그 / 주문 이력 / 현재 자산 상태는 UI에서 자주 조회하는 핵심 데이터이므로 즉시 조회 보존(hot)으로 영구 보존한다.
- 사이클 실행 로그는 최근 운영 분석과 실패 지점 추적을 위해 hot 90일을 유지한 뒤 cold로 이관해 영구 보존한다.
- audit log / 운영 이벤트 로그는 영구 보존 대상이지만, 주된 목적이 감사·운영 추적이므로 보관 저장(cold) 중심으로 관리한다.
- 설정 이력성 데이터(감시 종목 매핑 이력, 인스턴스 프롬프트 버전 이력)는 최근 운영 변경 확인을 위해 hot 90일을 유지한 뒤 cold로 이관해 영구 보존한다.

데이터 정리 작업은 일일 1회 오프피크 시간대에 수행한다.

## 10. 정책 및 제약

### 10.1 모드와 권한

- 전략 인스턴스는 `paper`와 `live` 모드를 가질 수 있다.
- `live` 모드는 연결 계좌가 있는 인스턴스에서만 활성화할 수 있다.
- 메인 모니터링 화면은 비로그인 조회를 허용한다(§4.1, §6.2).
- 설정 변경과 모든 편집/저장 액션은 운영자만 가능하다.
- 전략 인스턴스 선택은 전역 상태로 유지한다.
- 전략 추가 UX는 단순해야 하며, 운영자가 기존 주기형 전략 템플릿을 복제하거나 기본값만 정해서 시작할 수 있어야 한다.
- 일부 고급 운영 기능은 API는 존재하지만 메인 UI에는 아직 연결되지 않을 수 있다.
- 손익 관련 수치는 사용자 모니터링 지표로 제공하며, 정산 공식 자체는 본 문서 범위에 포함하지 않는다.

### 10.2 인스턴스 lifecycle

현재 lifecycle 주상태는 `draft`, `active`, `inactive` 세 가지다.

- `draft`, `active`, `inactive` 사이 전이는 모두 허용한다.
- 인스턴스 삭제/보관 정책은 본 버전 범위에서 정의하지 않는다.
- 비활성 시 자동 청산은 수행하지 않는다. 보유종목/잔고는 읽기 전용 조회만 가능하며 미청산 보유가 있으면 경고를 표시한다.
- `draft`는 미완성 설정을 허용하는 준비 상태다. 이름, 연결 계좌, 프롬프트, 입력 스펙, 감시 종목, 모델, 예산 등이 일부 비어 있거나 검토 중이어도 저장할 수 있다.
- `active` 전이는 "생성 가능"이 아니라 "실행 가능" 검증을 통과한 경우에만 허용한다.
- `draft → active`, `inactive → active` 전이 전에는 최소 다음 조건을 만족해야 한다.
  - 유효한 트레이딩 판단 프롬프트가 존재할 것
  - 유효한 트레이딩 모델이 결정되어 있을 것
  - 유효한 전략 입력 스펙이 결정되어 있을 것
  - 유효한 인스턴스 실행 주기가 결정되어 있을 것
  - 전략 예산이 설정되어 있을 것
  - `live` 모드라면 연결 계좌가 선택되어 있을 것
  - 평가 대상 범위가 `full_watchlist`라면 감시 종목이 1개 이상일 것
- 위 조건의 값은 템플릿 기본값 또는 인스턴스 개별 설정 중 실제 실행 시 적용될 유효값으로 판단한다.
- `active` 상태에서 허용되는 편집: 프롬프트(저장 시 신규 버전 생성), 전략 입력 스펙, 감시 종목, 모델, 인스턴스 단위 시스템 파라미터 개별 설정. 모두 다음 사이클부터 반영한다.
- `draft` / `inactive`에서만 변경 가능: 인스턴스명, 연결 계좌, 실행 모드(`paper / live`).
- `전략 예산`은 `draft`에서만 변경 가능하다.
- `inactive → active` 재활성화는 reset이 아니라 resume으로 간주한다. 기존 현금, 보유 종목, 손익 상태는 그대로 이어지며 자동 초기화하지 않는다.
- 복제는 설정만 복제한다(프롬프트, 전략 입력 스펙, 감시 종목, 모델, 시스템 파라미터, 예산, 모드, 연결 계좌). 자산/보유/판단 로그/주문 이력은 복제하지 않으며 복제 결과는 항상 `draft` 상태로 시작한다.
- `auto_paused`는 lifecycle 주상태가 아니라 `active` 위에만 얹히는 별개의 운영 메타 상태다. 쉽게 말해 "자동 일시중지" 상태다. 필드는 `auto_paused_reason`이며 허용 사유는 `reconcile_failed`다.
- `auto_paused_reason != null`인 인스턴스는 여전히 `active`로 간주하지만, 스케줄러는 다음 신규 사이클을 시작하지 않는다.
- `draft` / `inactive` 상태에서는 `auto_paused_reason`을 가질 수 없으며, 해당 상태로 전이될 때 값은 즉시 비운다.

### 10.3 활성 인스턴스 상한 및 비용 가드

- 활성 인스턴스 수 상한은 시스템 파라미터다. 권장 기본값 3, 하드 상한 10.
- 인스턴스별 일일 LLM 비용 상한은 시스템 파라미터로 관리한다.
- 활성 인스턴스 수 상한 계산에는 `auto_paused` 인스턴스도 포함한다. 자동 일시중지를 활성 인스턴스 cap 우회 수단으로 사용하지 않는다.
- 인스턴스별 일일 추정 비용이 상한을 초과하면 해당 사이클 종료 후 운영 알림을 준다.

### 10.4 live 모드 정책

- 연결 계좌는 §9의 entity로 관리하며, 시점상 1:1로 활성 live 인스턴스와 매핑된다.
- live 실주문은 본 spec의 범위에 포함된다(브로커 API의 세부 프로토콜은 별도 하위 spec).
- `client_order_id`는 시스템이 각 실제 주문 제출(`Order`)마다 신규로 발급하는 고유 식별자이며, 브로커 요청과 내부 주문 이력을 연결하는 추적 키로 사용한다.
- 판단과 주문은 1:N 관계로 분할 주문을 허용한다.
- 주문 상태 동기화(reconcile) 절차: 다음 사이클 진입 전 미확정 주문 상태를 다시 조회해 내부 기록과 맞춘다.
- 주문 상태 동기화에 실패하면 인스턴스를 `auto_paused(reason=reconcile_failed)`로 전환한다. 이 상태에서는 운영자 해제 전까지 신규 live 주문 사이클을 시작하지 않는다.
- 주문 상태머신은 `paper`/`live` 공통으로 `requested / accepted / partial / filled / canceled / rejected / failed`를 사용한다. 부분체결은 누적 체결 수량을 기록한다.

### 10.5 paper 모드 단순화 가정

- 호가창은 시뮬레이션하지 않는다.
- `paper`도 `live`와 같은 주문 상태머신을 사용한다.
- 주문가에 즉시 전량 체결되며 부분체결은 시뮬레이션하지 않는다. 일반적인 정상 흐름은 `requested → accepted → filled`다.
- `MARKET` 주문의 체결가는 주문 제출 시점의 최신 현재가 스냅샷으로 결정한다. `LIMIT` 주문은 LLM이 제안한 `price` 값으로 즉시 전량 체결된 것으로 처리한다.
- 슬리피지는 기본값 0.
- 수수료는 글로벌 운영 설정의 단일값을 적용한다. 이 값은 `paper` 모드에만 적용하며 `live` 모드에는 사용하지 않는다.
- "전략 예산"은 `draft` 상태 인스턴스가 최초로 `active` 전이될 때 초기 현금 잔고로 사용한다.
- `inactive → active` 재활성화 시에는 기존 현금, 보유 종목, 손익 상태를 그대로 유지하며 전략 예산으로 다시 초기화하지 않는다.
- 자산/보유/손익은 인스턴스 단위로 분리 추적하며, 동일 종목을 여러 인스턴스가 독립 보유할 수 있다.

### 10.6 회계 (모니터링 표시용)

- 보유 종목 평단은 이동가중평균법으로 산정한다.
- 평가손익 = (현재가 − 이동가중평균 매수단가) × 현재 보유수량
- `paper` 모드 실현손익 = (매도가 − 이동가중평균 매수단가) × 매도수량 − paper 전용 수수료
- `live` 모드 실현손익은 브로커 체결/정산 기준 값을 그대로 따른다.
- 수익률 분모 = 이동가중평균 매수단가 × 현재 보유수량
- 정확한 세금/정산 공식 자체는 §10.1 정책에 따라 본문 범위 외다.

### 10.7 시간 표기

- 서버와 DB의 시각 저장은 KST 기준으로 처리한다.
- "오늘"과 거래일 기준은 KST 00:00:00 ~ 23:59:59를 의미한다.
- UI 시간 표기는 `YYYY-MM-DD HH:mm KST` 형식으로 고정한다.
- 업무 날짜 계산과 보존/집계 기준에는 `business_date`를 사용한다.

### 10.8 보안 NFR

- live 계좌번호 등 식별 정보는 UI에서 마스킹한다.
- HTTPS를 강제하고 HSTS를 적용한다.
- 민감 secret(브로커 API 키, JWT secret 등)은 GitHub Secrets로 관리하고, 배포 시 서버의 `.env`로 주입하며 평문 저장을 금지한다.
- 운영 환경에서는 debug 모드를 비활성화하고 stack trace를 노출하지 않는다.

### 10.9 운영 알림

- 운영 알림 채널은 Telegram으로 고정하며, on/off 및 음소거 시간대를 지원한다.
- 운영 알림은 오류가 발생했을 때 즉시 발송한다.
- 대상 예: 시장 데이터 조회 실패, 계좌 조회 실패, 주문 API 실패, 판단 로그 `FAILED`, 일일 추정 비용 상한 초과, `auto_paused(reason=reconcile_failed)`, 활성 인스턴스의 다음 사이클 진입 자체가 불가능한 실행 오류
- 같은 원인으로 반복 발생하는 알림은 원인별로 묶어 전송한다. 동일 원인이 계속 열려 있는 동안에는 상태 변화(발생/복구)가 있을 때만 추가 발송한다.

## 11. 전략 템플릿 및 인스턴스 추가 설계 원칙

새 주기형 전략을 쉽게 추가하기 위해 아래 원칙을 따른다.

- 생성 가능 조건과 활성화 가능 조건은 분리한다. 생성은 `draft` 저장 기준이고, 활성화는 실제 실행 가능 기준이다.
- 전략 템플릿 생성 시 최소 입력값은 `전략명`, `설명`, `기본 실행 주기`다.
- 템플릿은 위 최소값만으로 먼저 생성할 수 있다. 다만 프롬프트, 기본 모델, 기본 입력 스펙이 비어 있으면 미완성 템플릿으로 간주한다.
- 전략 인스턴스 생성 시 최소 입력값은 `인스턴스명`, `실행 모드`, `전략 예산`이다.
- 인스턴스도 위 최소값만으로 `draft` 생성할 수 있다. 프롬프트, 모델, 입력 스펙, 감시 종목, 연결 계좌가 일부 비어 있어도 `draft` 저장은 허용한다.
- `live` 인스턴스는 활성화 시점에는 연결 계좌가 전제되어야 한다.
- 인스턴스 활성화 가능 조건은 §10.2 lifecycle의 활성화 전 체크리스트를 따른다.
- 값 존재 여부는 인스턴스 개별 설정이 없더라도 템플릿 기본값이나 시스템 기본값을 통해 실제 실행값이 결정되면 충족한 것으로 본다.
- 전략 인스턴스 생성 시 선택적으로 기존 템플릿 또는 기존 인스턴스의 종목, 프롬프트, 전략 입력 스펙, 인스턴스 실행 설정을 복제할 수 있어야 한다(자산/판단 로그/주문 이력은 복제하지 않는다).
- 전략별 프롬프트는 인스턴스 종속이며, 인스턴스 생성 시 템플릿에서 복사한 후 독립 버전 이력을 가진다.
- 인스턴스별 감시 종목은 글로벌 마스터를 참조한 인스턴스별 매핑으로 독립 관리된다.
- 모델 설정과 전략 입력 스펙은 글로벌 공통값이 아니라 템플릿 기본값 + 인스턴스별 개별 설정 구조를 가진다.
- 인스턴스 실행 설정은 인스턴스 운영값으로 관리한다.
- 글로벌 운영 설정은 시스템 전체에 단일값으로 적용되며 템플릿/인스턴스별 개별 설정 대상이 아니다.
- 운영자는 새 템플릿이나 인스턴스를 만든 뒤 즉시 활성화하지 않고 검토 후 활성화할 수 있어야 한다(생성 직후 상태는 항상 `draft`).

## 13. 비기능 요구사항 (NFR)

### 13.1 데이터 일관성

- 설정 저장은 `updated_at` / `version` 기반 낙관적 잠금(optimistic lock)을 적용해 동시 수정 충돌 시 재시도를 강제한다.
- 사이클 실행은 시작 시점에 프롬프트/모델/전략 입력 스펙/인스턴스 실행 설정/감시종목 목록을 고정 사용하며, 도중에 변경된 설정은 다음 사이클부터 반영한다.

### 13.2 판단 JSON 스키마

LLM이 실제로 실행된 판단 로그의 출력은 다음 스키마를 따른다. `cycle_status`에 따라 최소 필드는 다음과 같다.

`EXECUTE`일 때 최소 필드:

- `cycle_status`: `EXECUTE`
- `summary`: 사이클 수준 판단 요약
- `confidence`: 모델 자체 신뢰도 (0.0 ~ 1.0)
- `orders[]`: 실행 제안 목록

`HOLD`일 때 최소 필드:

- `cycle_status`: `HOLD`
- `rationale`: 사이클 수준 판단 근거 텍스트
- `confidence`: 모델 자체 신뢰도 (0.0 ~ 1.0)
- `evidence[]`: 사이클 수준 근거 목록
- `orders[]`: 빈 배열

`orders[]`의 각 항목은 최소 다음 필드를 가진다.

- `symbol`: 종목코드
- `side`: `BUY` / `SELL`
- `quantity`: 주문 수량
- `order_type`: `MARKET` / `LIMIT`
- `price`: nullable 주문가 (`MARKET`일 때만 `null` 허용)
- `rationale`: 해당 주문의 근거
- `evidence[]`: 해당 주문에 직접 연결된 근거 목록

`evidence[]`의 각 항목은 최소 다음 필드를 가진다.

- `source_type`: 근거 데이터 소스(source) 종류
- `summary`: 근거 요약
- `contribution`: 판단에 어떤 영향을 줬는지 설명

`HOLD`/no-action은 빈 `orders[]`만으로 암묵 처리하지 않고 `cycle_status=HOLD`로 명시해야 한다. `EXECUTE`인 경우에는 사이클 수준 `summary`를 포함하고, 실제 근거 설명은 각 주문의 `rationale`과 `evidence[]`에 둔다. 출력이 구조적으로 깨졌거나, 필수 필드 누락/타입 오류/허용되지 않은 값/의미 충돌 같은 내용 검증에 실패하면 해당 판단 로그는 `FAILED`로 기록하며 부분 수용하지 않는다. 내용 검증을 통과한 후에도 주문 제출 직전 최소 안전검증(현금 범위, 보유 수량 범위, `order_type`/`price` 일관성 등)에 실패하면 해당 판단의 주문 제출은 수행하지 않고 검증 실패로 기록한다. `auto_paused`는 LLM 출력 상태가 아니라 시스템 운영 메타 상태이며, 쉽게 말해 "자동 일시중지"를 뜻한다. 판단 로그에는 사용 모델, 입력 토큰, 출력 토큰, 추정 비용과 판단 시점 설정값을 함께 기록하며 §7.3 판단 상세에서 함께 표시한다.

| Telegram Bot API | 운영 알림 발송 | `https://api.telegram.org/bot<TOKEN>/sendMessage` | `TELEGRAM_BOT_TOKEN`, `TELEGRAM_CHAT_ID` | 트레이더 에러, 이벤트 트레이더, 일일 리포트, 배포 알림 |

추가 메모:

- 트레이딩 판단용 LLM 호출은 현재 `openclaw` / `nanobot` CLI를 subprocess로 실행하는 구조이며, 이 코드는 공개 HTTP API 엔드포인트를 직접 하드코딩하지 않는다.
- GitHub webhook은 외부 API를 호출하는 outbound 연동이라기보다, `/api/deploy`가 외부에서 서명된 요청을 받는 inbound 연동이다. 인증값은 `WEBHOOK_SECRET`이다.
